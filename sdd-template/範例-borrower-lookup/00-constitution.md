# 專案憲章 (Constitution)

> 這份文件定義整個專案**永遠**要遵守的規矩。
> 每次叫 AI 寫程式前，都讓它先讀這份。一個專案只需要一份，長期沿用。

---

## 1. 專案一句話

給風管部同仁使用的內部小工具：輸入授信戶統一編號，查出該戶目前的風險分級與最後更新日。

## 2. 技術選型 (Tech Stack)

| 類別 | 選用 | 備註 |
|------|------|------|
| 語言 | Python 3.12 | |
| 框架 | FastAPI | 後端 API |
| 前端 | 純 HTML + Tailwind（CDN 引入） | 不引進前端框架，保持簡單 |
| 資料庫 | SQLite | 單機內部工具，免架資料庫伺服器 |
| 測試 | pytest | |
| 套件管理 | uv | |

## 3. 程式風格 (Coding Style)

- 命名：變數 / 函式 snake_case、類別 PascalCase。
- 註解：公開函式一律寫 docstring，用繁體中文。
- 檔案長度：單檔超過 300 行就要拆。
- 格式化工具：ruff format，提交前一定跑。

## 4. 架構原則 (Principles)

- 商業邏輯與 I/O 分離：邏輯層不直接寫 SQL，透過資料層存取。
- 對外輸入一律驗證：統編格式先驗證再查詢。
- 能用標準函式庫就不裝第三方套件。
- 每個功能都要有對應 pytest 測試。

## 5. 絕對禁止 (Never Do)

- ❌ 不要把資料庫路徑、任何金鑰寫死在程式碼裡，用設定檔或環境變數。
- ❌ 不要為了通過測試而修改測試本身。
- ❌ 憲章沒列出的套件不要自己裝。
- ❌ 不要一次改很多檔案，一次專注一個任務。
- ❌ 不確定就停下來問，不要腦補需求。

## 6. AI 協作規範

- 動工前先讀：本憲章 + 該功能的 `spec` 與 `design`。
- 每完成一個任務就停下來，讓我確認後再繼續。
- 改動超出當前任務範圍前，先說明原因並徵得同意。
- 回報時老實講：測試沒過就說沒過，跳過的步驟要講。
