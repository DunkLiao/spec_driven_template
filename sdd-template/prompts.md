# 提示詞範本 (Prompts)

> 每個階段可直接複製貼上的提示詞。把 `[功能名]` 換成你的功能即可。
> 每個新對話的第一句，都先叫 AI 讀相關檔案。

---

## ① 從想法產生規格

```
我要做一個新功能：[一句話描述你的想法]。

請先讀 00-constitution.md，然後照 01-spec.md 的範本，幫我把這個想法寫成一份規格草稿。
規則：
- 只談「做什麼」，不要寫技術實作。
- 不確定的地方放進「待確認問題」，不要自己腦補。
- 寫完後列出三個你覺得我最需要先決定的問題。
```

## ② 規格 → 技術設計

```
請讀 00-constitution.md 和 01-spec-[功能名].md。
照 02-design.md 的範本，幫我寫這個功能的技術設計。
規則：
- 不可違反憲章的技術選型與禁止事項。
- 每個關鍵決策都要寫「為什麼選它、放棄什麼」。
- 確認每一條驗收標準 (AC) 在設計裡都有對應的處理方式。
```

## ③ 設計 → 任務拆解

```
請讀 02-design-[功能名].md。
照 03-tasks.md 的範本，把工作拆成可逐項打勾的小任務。
規則：
- 每個任務 15～30 分鐘內可完成、可獨立驗證。
- 照相依順序排列。
- 每個任務標明「完成的定義」，涉及測試的標上對應 AC 編號。
```

## ④ 逐項實作（最常用）

```
請讀 00-constitution.md、02-design-[功能名].md、03-tasks-[功能名].md。
從 03 裡第一個未打勾的任務開始做。
規則：
- 一次只做這一個任務，不要順手改別的。
- 做完後：把該任務打勾、簡述你改了什麼、告訴我怎麼驗證。
- 然後停下來等我確認，不要自己往下一個任務衝。
```

## ⑤ 實作中發現規格不對

```
我在做 [任務] 時發現規格有問題：[說明問題]。
請先不要改程式。
先回去更新 01-spec-[功能名].md（或 02-design），把變更和原因寫清楚，
等我確認新規格後，我們再回來改程式。
```

## ⑥ 階段審查 / 收尾

```
請對照 01-spec-[功能名].md，逐條檢查驗收標準 (AC) 是否都已滿足。
列成一個表：AC 編號 | 狀態(過/未過/未驗) | 對應的測試或程式位置。
未過的，告訴我缺什麼。不要為了讓它看起來過而修改測試。
```

---

## 小提醒

- AI 一次記不住太多 —— 與其貼一大段，不如叫它去讀 `.md` 檔。
- 它「忘記」規格時，回到 ① 重貼讀檔指令即可。
- 你的角色是**審查者**：每個階段產出後先審再往下。
