# 規格驅動開發 (Spec-Driven Development, SDD) 範本

> 給 vibe coding 用的「先寫規格、再讓 AI 寫程式」工作流程。
> 核心理念：**AI 寫出來的東西好不好，取決於你規格寫得清不清楚。**

---

## 為什麼要 SDD？

直接叫 AI「幫我做一個 XXX」常常會：

- 越改越亂，改 A 壞 B
- AI 自己腦補需求，做出你沒要的東西
- 換一個對話視窗、換一個工具，前面講過的全忘光

SDD 的做法是把「你腦中的想法」先固化成幾份規格文件，當作**單一事實來源 (Single Source of Truth)**。
之後不管哪個 AI、哪個對話，都先讀這幾份規格再動工。

---

## 四個檔案、四個階段

| 順序 | 檔案 | 階段 | 回答的問題 |
|------|------|------|-----------|
| 0 | `00-constitution.md` | 憲章 | **規矩**：這專案永遠要遵守哪些原則？(技術選型、風格、禁止事項) |
| 1 | `01-spec.md` | 規格 | **做什麼**：要解決什麼問題？使用者要什麼？怎樣算做完？ |
| 2 | `02-design.md` | 設計 | **怎麼做**：用什麼架構、資料怎麼存、模組怎麼切？ |
| 3 | `03-tasks.md` | 任務 | **照什麼順序做**：拆成一個個小步驟，逐項打勾完成。 |

> 憲章只寫一次、長期沿用；規格 / 設計 / 任務每個功能 (feature) 各寫一份。

---

## 標準工作流程

```
① 寫憲章 (一次性) ──▶ ② 寫規格 ──▶ ③ 寫設計 ──▶ ④ 拆任務 ──▶ ⑤ 讓 AI 逐項實作
        00              01            02            03            (照 03 的清單)
                         │            │             │
                         └─ 每一步都讓 AI 先讀前面的檔案，確認無誤再往下 ─┘
```

實際操作（搭配 Claude Code / Cursor / 任何 AI coding 工具）：

1. **開新功能** → 複製 `01`、`02`、`03` 三個範本，依需要改檔名（例：`spec-login.md`）。
2. **填規格** → 自己先填 `01-spec.md`，或把想法丟給 AI、叫它照範本幫你補完，你再審。
3. **產設計** → 對 AI 說：「讀 `00-constitution.md` 和 `01-spec.md`，照 `02-design.md` 範本幫我寫技術設計。」審過再繼續。
4. **拆任務** → 對 AI 說：「照 `02-design.md`，把工作拆成 `03-tasks.md` 那種可逐項打勾的小任務。」
5. **逐項實作** → 對 AI 說：「照 `03-tasks.md`，從第一個未完成任務開始做，做完一個停下來讓我確認。」

`prompts.md` 裡有每個階段可以直接複製貼上的提示詞。

---

## 黃金守則

- **一次只往前一步**：規格沒定好不要碰設計；設計沒定好不要碰程式。
- **AI 動工前先讀規格**：每個新對話的第一句都叫它先讀相關的 `.md`。
- **規格是活的**：實作中發現規格不對，先回去改規格，再改程式 —— 不要只改程式。
- **不確定就標 `[待確認]`**：寧可留問號，不要讓 AI 腦補。
- **你是審查者**：每個階段 AI 產出後，你的工作是審、不是照單全收。
