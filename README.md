# Spec-Driven Template（規格驅動開發範本）

簡介
----
這個專案提供一套「規格驅動開發（Spec-Driven Development, SDD）」的範本與流程，用於把想法先固化成可驗證的規格、再讓 AI 或人員逐步實作。重點：規格是唯一事實來源，程式必須服從規格。

專案結構（重要目錄/檔案）
-------------------------
- sdd-template/
  - README.md                範本說明（工作流程與範例）
  - CLAUDE.md                專案 AI 協作守則（請 AI 每次讀取）
  - 00-constitution.md       專案憲章（技術選型、禁止事項）
  - 01-spec-*.md             每個功能的需求與驗收標準
  - 02-design-*.md           每個功能的技術設計
  - 03-tasks-*.md            每個功能的任務清單（逐項實作）
- sdd/                       其他參考與用例
- sdd-review/                審查或範例實作

為什麼用 SDD？
--------------
- 減少 AI 或開發者的腦補行為
- 將需求化為明確、可驗證的驗收標準（AC）
- 支持可重複、可審查的 AI 協作流程

核心工作流程（簡要）
---------------------
1. 撰寫/檢視 `00-constitution.md`（每次必看）
2. 撰寫/確認 `01-spec-*.md`（需求與 AC）
3. 產出 `02-design-*.md`（技術設計）
4. 拆成 `03-tasks-*.md`（小任務、可打勾）
5. 逐項由 AI/開發者執行：每次只做一個未完成任務，完成後停下並回報驗收狀態

安裝與使用（範本專案）
--------------------
這個倉庫為工作流程與範本，通常不需要安裝程式碼。使用方式：

1. 複製範本檔案建立新功能：
   - 建立 `01-spec-<feature>.md`：描述需求、使用者場景、驗收標準（AC）
   - 建立 `02-design-<feature>.md`：技術設計、資料結構、API 介面（如適用）
   - 建立 `03-tasks-<feature>.md`：把工作拆成可打勾的小任務

2. 與 AI 協作舉例：
   - 每次新對話第一句要先讓 AI 讀取 `00-constitution.md` 與該 feature 的 `01/02/03` 檔案
   - 指示 AI 從 `03` 的第一個未完成任務開始，完成一項就停，回報驗收結果

範例：簡單的 spec 範本（01-spec-login.md）
```markdown
# 01-spec-login.md

目標
- 使用者可以用 email 與密碼登入

驗收標準（AC）
- AC1: 成功登入後回傳 200 與使用者基本資訊
- AC2: 輸入錯誤憑證時回傳 401
- AC3: 密碼不得以明文儲存，需使用安全雜湊
```

範例任務（03-tasks-login.md）
- [ ] T1: 建立使用者模型（含密碼雜湊） — 對應 AC3
- [ ] T2: 實作登入 API 與驗證流程 — 對應 AC1/AC2
- [ ] T3: 撰寫測試覆蓋 AC1/AC2/AC3

AI 協作守則（要點）
-------------------
- 每次工作前先讀 `00-constitution.md`。
- 一次只做一個 `03` 任務；完成後回報對應 AC 是否達成並說明驗證步驟。
- 不確定時標記 `[待確認]` 並等待人工確認，切勿自行腦補需求。

驗證與測試
---------
- 每個 feature 的 `03` 任務應包含測試或手動驗證步驟（如何呼叫 API、輸入範例、期望回應）。
- 任務完成後附上測試輸出或驗證截圖／步驟說明。

貢獻指南
--------
- 修改或新增範本請依專案憲章（`00-constitution.md`）進行。
- 發 PR 或 Issue 時請說明對應的 `01/02/03` 檔案變動與驗證方式。
