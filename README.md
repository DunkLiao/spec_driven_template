# Spec-Driven Template（規格驅動開發範本）

簡短說明
--------
這個專案提供一套簡單、可重複的「規格驅動開發（SDD）」工作流程。流程核心：先寫清楚可驗證的規格，再依規格一步步實作與測試。這能減少誤解、方便與 AI 或團隊合作。

適合對象
--------
- 開發者與產品人員，想把需求變成明確、可測的工作項目
- 與 AI（如程式生成或測試工具）協作的團隊

快速上手（3 個步驟）
------------------
1. 寫規格：建立 `01-spec-<feature>.md`，描述目標、使用情境與驗收標準（AC）。
2. 設計：建立 `02-design-<feature>.md`，說明資料結構、API 與實作重點（若需要）。
3. 拆任務：建立 `03-tasks-<feature>.md`，把工作拆成可執行的小任務，每次只做一項並驗證對應的 AC。

常見檔案與目錄
----------------
- `00-constitution.md`：專案憲章（必看，包含風格、約束與禁止事項）
- `01-spec-*.md`：功能需求與驗收標準
- `02-design-*.md`：技術設計與介面定義
- `03-tasks-*.md`：逐項任務列表，含驗證步驟或測試說明

AI 協作建議
----------
- 每次開始新工作前，先讓 AI 讀取 `00-constitution.md` 與該功能的 `01/02/03` 檔案。
- 指示 AI 從 `03` 的第一個「未完成」任務開始：完成一項就停止，回報是否滿足對應的 AC，並說明驗證步驟與輸出。
- 若 AI 有不確定處，要求標註為 `[待確認]` 並等待人工確認。

範例（簡短）
---------
01-spec-login.md 中可能包含：
- 目標：使用者可以用 email 与密碼登入
- 驗收標準（AC）：成功登入回傳 200、錯誤憑證回傳 401、密碼以雜湊儲存

貢獻方式
--------
- 修改前請閱讀 `00-constitution.md`。
- 提交 PR 時，在描述中說明修改的是哪個 `01/02/03` 檔案，以及如何驗證變更。
